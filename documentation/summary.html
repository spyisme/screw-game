Screw C++ Client Project Summary

This document describes the current C++/SFML frontend client for the multiplayer card game Screw. The Flask backend owns rooms, players, cards, turns, validation, hidden-card visibility, and game rules. The C++ client handles rendering, input, simple animations, texture caching, and HTTP calls.

Project Overview

The project is a Windows Visual Studio C++ application using SFML 3 for graphics. It connects to the Flask backend using values from config.json. The app window title and UI use the game name Screw. The default launch window is 1000x700, and the packaged config can override the launch width and height.

The active gameplay client no longer uses local deck drawing or local player-hand mutation as the source of truth. Instead, it reads the latest JSON state from the backend and sends action requests when the player clicks the deck, discard pile, or a hand card.

Current User Flow

  1. The app starts on the main menu.
  2. Clicking Start Game asks for a nickname.
  3. After nickname entry, the client calls GET /rooms and shows public rooms.
  4. The player can click a room to join, click Refresh Rooms, or click Create New Room.
  5. The waiting room shows the room and player name.
  6. The host sees the room code, joined player names, and a count such as 2/4.
  7. The host waiting room polls GET /rooms/<room_id>/lobby every 2 seconds to update player names and count.
  8. The host sees the Start button and can start the game with Enter or by clicking Start.
  9. Players who join an existing room do not see the Start button. They see Waiting for host to start....
  10. Non-host players poll the backend every 2 seconds while waiting for the host to start.
  11. When the game starts, the client renders the player-specific backend state and shows the current turn at the top.
  12. During play, every client polls game state every 2 seconds.
  13. Opponent cards are rendered at the same size as the local player's cards, so all players read at one visual scale.
  14. Remote player discard animations now infer the source from state changes: deck/card-pile discards animate from the deck pile, pending drawn cards animate from the drawn slot, and hand-origin actions animate from the opponent row.
  15. The game table uses highlighted card outlines to show possible deck, discard, and hand-card actions.

UI/UX State

  • The main menu now uses a table-style framed scene, centered title, subtitle, hover-aware Start and Exit buttons, and clearer visual hierarchy.
  • The nickname screen now has a framed panel, title, input line, and Enter hint instead of a single raw text line.
  • The room browser uses a framed layout, stronger button styling, hover outlines, and better empty-room text.
  • The waiting room is framed and separates host controls from joined-player waiting state.
  • The gameplay screen is divided into opponent area, central table area, and player hand area.
  • The deck, discard pile, drawn-card slot, and hand cards have labels and hover/action outlines.
  • Self and opponent cards share the same CARD_SIZE, currently 96x132, with common spacing constants for predictable layout edits.
  • Long UI strings are fitted before drawing so player names, room rows, hints, current-turn labels, and status messages do not overlap surrounding controls.
  • Actionable card slots use green or gold outlines when the backend says the corresponding action is allowed.
  • Remote actions use eased ghost-card movement with a shadow and highlight outline, matching the table layout instead of appearing as a disconnected jump.
  • Status and error messages are shown in a bottom status bar instead of floating over the top-left of the play field.
  • The SFML view is fixed to a 1000x700 logical canvas, so custom launch sizes from config scale the same UI layout.

Runtime Config

The executable reads config.json from the working directory at startup, so the backend host and port can be changed without recompiling. 

{
  "host": "127.0.0.1",
  "port": 5000,
  "width": 1000,
  "height": 700
}

The packaged build includes final/config.json beside final/Skrew.exe. The source project also includes Skrew/config.json for running from the project folder. If the config file is missing or invalid, the client falls back to 127.0.0.1:5000 and 1000x700. If width or height is missing, not an integer, or less than 1, that value falls back to the default.

Source Files

Skrew/src/main.cpp Main application entry point. Creates the SFML window, owns the app state machine, handles menu, nickname input, room discovery, waiting room, gameplay input, polling, animation, and rendering. Reads config.json, then reads cards, discard, pending drawn card, current turn, lobby players, allowed actions, and launch window size from backend/config JSON. Skrew/src/Deck.cpp Legacy/local deck implementation. It remains in the project, but active gameplay should not use it as the source of truth because the backend owns real game logic.
Skrew/src/Player.cpp Legacy/local player implementation. It remains in the project, but active gameplay should not mutate local player hands because backend state is authoritative.

Backend Endpoints Used

Endpoint Client Method Purpose
POST /rooms GameClient::createRoom Create a new public room.
GET /rooms GameClient::fetchRooms Fetch public rooms for the room discovery page.
POST /rooms/<room>/join GameClient::joinRoom Join an existing room or newly created room.
GET /rooms/<room>/lobby GameClient::fetchLobby Poll waiting-room player names and player count for the host lobby display.
POST /rooms/<room>/start GameClient::startGame Host starts the game.
GET /rooms/<room>/state?player_id=... GameClient::refreshState Fetch player-specific game state for rendering.
POST /rooms/<room>/actions drawFromDeck, dropDrawnCard, swapDrawnWithOwnCard, takeDiscardAndSwap Send gameplay actions to the backend.

Backend JSON Fields Rendered by Client

  • players[].cards[]: player card slots.
  • face_down: hidden cards render as back.png.
  • card.image_name: visible cards render using this image name.
  • pending_drawn_card: drawn card waiting for swap/drop decision.
  • top_discard_card and top_discard: discard pile top card.
  • allowed_actions: client checks this before sending gameplay actions.
  • current_player_nickname: displayed at the top as the current player turn.
  • current_player_id: fallback for the current turn label when no nickname is provided.
  • you.nickname: player label fallback after game state is available.
  • room_code: displayed in the waiting room instead of the internal room ID.
  • players[].nickname: used for lobby player names and opponent labels.
  • players[].cards: own cards are rendered from state; opponent cards are shown as backs.
  • max_players: used with lobby player count for displays such as 2/4.
  • status and phase: included in the state signature used to detect remote action animation changes.
  • draw_pile_count: included in the state signature when available.
  • deck_count, deckCount, cards_pile_count, and cardsPileCount: fallback draw/card-pile count names used to detect whether a remote discard came from the card pile.

Important Notes

  • The backend host and port are read from config.json beside the executable.
  • The default backend config is 127.0.0.1:5000.
  • The launch window size is read from width and height in config.json; missing or invalid values fall back to 1000x700.
  • Use 127.0.0.1 instead of localhost if Windows IPv6/IPv4 resolution causes connection issues.
  • The host cannot start the game until at least two players have joined, according to the backend rules.
  • Only the host sees the waiting-room Start button. Joined players wait for the host and poll for game start.
  • The host lobby polls every 2 seconds to show updated joined player names and count.
  • The playing screen polls every 2 seconds to keep all clients synchronized.
  • The gameplay screen shows the current turn from current_player_nickname or current_player_id.
  • Remote action animation is visual-only and based on polling differences, because the backend state does not expose a full action event log. The client compares previous and refreshed state to choose deck, drawn-slot, or opponent-row animation sources.
  • The new UI highlights only actions inferred from allowed_actions; backend validation remains the source of truth.
  • The active gameplay renderer depends on GET /rooms/<room_id>/state?player_id=....
  • Legacy Deck and Player classes remain for now but should not become gameplay authority again.
