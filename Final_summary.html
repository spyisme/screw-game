Skrew Codebase Final Summary

This file explains the whole repository: what the application is, how it runs, how the main systems work, and what each tracked project file or packaged file is responsible for.

Overall Purpose

Skrew is a Windows C++ desktop client for a multiplayer card game named Screw/Skrew. It uses SFML 3 for the graphical interface, cpp-httplib for HTTP calls, and nlohmann/json for JSON parsing. The client does not own the real multiplayer rules. A separate backend at the configured host and port owns rooms, players, turns, hidden-card visibility, card movement, scoring, and allowed actions.

The executable starts a graphical window, lets the user enter a nickname, browse or create rooms, wait in a lobby, and then play cards by clicking the deck, discard pile, own cards, opponent cards, and the Screw button. The client repeatedly polls the backend and redraws the latest JSON state.

Architecture

UI and app state Skrew/src/main.cpp contains the SFML app loop, screen state machine, rendering helpers, click handling, animations, and JSON interpretation helpers.
Backend client GameClient wraps all HTTP calls to the backend, stores the room ID, room code, player ID, latest game state, and last error.
Assets Card images live in Skrew/assets/cards, and the UI font is Skrew/assets/fonts/arial.ttf. The release package copies those assets into final/dlls/assets.
Build and package Visual Studio builds Skrew.vcxproj. Release x64 runs package_final.ps1, which creates final/ and final.zip.

Runtime Flow

  1. main() configures DLL search paths, catches fatal errors, and calls runApp().
  2. runApp() loads config.json, opens the SFML window, loads the font, and creates UI state.
  3. The app begins in AppState::Menu, drawn by MainMenu.
  4. Starting the game moves to nickname entry, then room selection.
  5. The client fetches rooms with GET /rooms, creates rooms with POST /rooms, and joins with POST /rooms/<id>/join.
  6. The waiting room polls GET /rooms/<room_id>/lobby. The host can start with POST /rooms/<room_id>/start.
  7. The playing screen polls GET /rooms/<room_id>/state?player_id=... and renders the returned player-specific JSON.
  8. Gameplay clicks send actions through POST /rooms/<room_id>/actions, then refresh the state.
  9. Round-end data is displayed from results or round_results.

Main Gameplay Concepts

  • Number cards: images and values include -1, 0 through 10, 20, and 25.
  • Special cards: GiveAndTake swaps a local card with an opponent card; Seeandswab supports reveal/swap behavior.
  • Drawn card: a card drawn from the deck can be swapped into the player's hand or discarded.
  • Discard pile: the top discard can be taken and swapped when the backend allows it.
  • Screw action: calling Screw starts final turns. At round end, if the caller does not have the lowest score, a penalty is applied by the backend.
  • Allowed actions: the client checks allowed_actions for highlighting and click behavior, but backend validation remains authoritative.

Source And Header Files

FileFunctionWhat It Does
Skrew/src/main.cpp Main executable logic and SFML game client UI. Contains the app state machine, config loading, JSON parsing helpers, card visibility/image mapping, room and lobby screens, gameplay rendering, polling, click handling, animations, Screw confirmation, action reveal overlay, results screen, fatal error handling, and main().
Skrew/include/GameClient.h Declares the HTTP client class. Defines fields for backend host/port, room ID/code, player ID, latest state, last error, and public methods for room, lobby, state, and game actions.
Skrew/src/GameClient.cpp Implements backend communication. Uses cpp-httplib to send GET/POST requests, parses JSON responses, handles errors, creates and joins rooms, starts games, refreshes state, and sends actions such as draw, drop, swap, reveal, GiveAndTake, SeeAndSwap, cancel, and call Screw.
Skrew/include/MainMenu.h Declares the menu UI. Defines MenuResult and a MainMenu class with title, subtitle, Start button, Exit button, event handling, and drawing.
Skrew/src/MainMenu.cpp Implements the menu UI. Builds the menu text and buttons, detects mouse clicks, changes button colors on hover, and draws a table-like menu scene.
Skrew/include/TextureManager.h Declares texture caching. Defines a map of loaded SFML textures and exposes getTexture().
Skrew/src/TextureManager.cpp Loads card images. Looks first in dlls/assets/cards and then in assets/cards, loads each PNG once, smooths it, caches it by file name, and returns a pointer for rendering.
Skrew/include/Card.h Declares local card data. Defines CardType with Number, GiveAndTake, and SeeAndSwab, plus a simple Card struct with constructors for number and special cards.
Skrew/include/Deck.h Declares a local deck class. Defines a vector-backed Deck with create, shuffle, size, and draw operations. This is legacy/local logic; active multiplayer state comes from the backend.
Skrew/src/Deck.cpp Implements local deck behavior. Creates number cards from -1 to 10, high-value cards 20 and 25, special cards, shuffles with std::shuffle, and draws from the back of the vector.
Skrew/include/Player.h Declares a local player class. Stores a player name and vector of local Card objects with getters and hand mutation methods. This is also legacy/local data.
Skrew/src/Player.cpp Implements local player behavior. Constructs a player, gets/sets the name, adds cards, clears the hand, and returns the hand vector.
Skrew/include/third_party/httplib.h Bundled HTTP library. Single-header cpp-httplib dependency used by GameClient for HTTP requests to the backend.
Skrew/include/third_party/json.hpp Bundled JSON library. Single-header nlohmann/json dependency used for request bodies, response parsing, and state inspection.

Important Notes

  • The backend must be running at the configured host and port, defaulting to 127.0.0.1:5000.
  • The SFML UI is drawn on a fixed logical view of 1000x700, even if the launch window size changes.
  • Deck, Player, and Card are simple local model classes, but the active multiplayer game should use backend state as the source of truth.
  • The client intentionally accepts several alternative JSON field names, which makes it more tolerant of backend response changes.
  • Packaged DLLs and the final folder can be regenerated by a Release x64 build or by running package_final.ps1 with a built executable.
